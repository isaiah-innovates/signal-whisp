# When broadening your data sources breaks your eval's scope

My signal-extraction agent has one job: read a post, decide if it describes a real, recurring pain point worth tracking. It was passing its eval at a solid clip against a narrow pull — Stack Exchange's security site, tagged phishing, SIEM, vulnerability management, incident response. Every post in that pull was already security-adjacent by construction. The query was doing double duty as a topic filter, and the extraction prompt never had to know that.

Then I broadened the pull. More corroboration needed more volume, so I added ServerFault as a secondary site — monitoring, backup, patch-management tags, exactly as the project's own data-source notes suggested. More posts came in. More signals came out. And a few of them were... a mariadb backup process that hangs on busy databases. A single-server disaster-recovery plan with an unrealistic RTO. A mobile VPN client that can't hold two simultaneous tunnels.

Every one of those is a real, recurring, well-stated operational pain point. My extraction agent was completely correct to flag them as signals — nothing in its instructions said "and it must be a cybersecurity problem specifically," because that condition used to be free. The original query already filtered for security topics, so the prompt's actual bar (real pain, not venting, stated by a practitioner) never needed a security clause bolted on. Change the query, and the implicit filter goes with it. The explicit prompt, unchanged, does exactly what it always did — which is no longer what you want.

There's a tempting fast fix here: just add "must be cybersecurity-specific" to the extraction prompt and move on. I didn't do that, on purpose. Prompt changes are supposed to trace back to eval criteria, not the other way around — if I patch the prompt first and backfill the reasoning later, I've silently moved the goalposts of what my own eval file is testing for. So instead of quietly filtering or quietly patching, I flagged it as a real scope question, proposed a specific cluster-by-cluster prune with my reasoning for each, and got an explicit call before touching anything downstream.

The general shape of the trap: broadening your data sources in pursuit of more density or more corroboration can silently break scoping assumptions your extraction step was never asked to defend, because the query used to do that job for free. Every time you widen the net, it's worth asking what implicit filtering the old, narrower net was quietly providing — before you assume your extraction logic is still measuring the same thing it was yesterday.
